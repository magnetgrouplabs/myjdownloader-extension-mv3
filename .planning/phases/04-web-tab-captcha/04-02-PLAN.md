<objective>
Create the MYJD remote CAPTCHA solver content script, its supporting infrastructure (loginNeeded.html, manifest registration, background.js MYJD handlers), forming the foundation for Flow B.

Purpose: Users whose JDownloader runs on a NAS/server cannot use localhost CAPTCHA pages. Flow B renders CAPTCHAs on the target domain via a content script that replaces the page DOM, loads CAPTCHA widget scripts externally, and submits tokens through the MYJD cloud API.
Output: myjdCaptchaSolver.js content script, loginNeeded.html, updated manifest.json, updated background.js with MYJD handlers.

**MYJD token submission path note:** CONTEXT.md says "Token submitted via myjdDeviceClientFactory /captcha/solve API." The implementation routes solved tokens through chrome.tabs.sendMessage to my.jdownloader.org tabs (via webinterfaceEnhancer.js), which then calls the MYJD API on behalf of the extension. This IS the myjdDeviceClientFactory path -- the web interface tab has AngularJS loaded with myjdDeviceClientFactory, and it calls /captcha/solve when it receives the solution message. The service worker cannot call myjdDeviceClientFactory directly (no AngularJS context), so routing through the web interface tab is the established pattern (see Rc2Service.sendRc2SolutionToJd lines 86-106, RESEARCH.md Pitfall 5 and Open Question 2).
</objective>

**Create `contentscripts/myjdCaptchaSolver.js`** -- the MYJD remote CAPTCHA solver content script. Runs at `document_start` on all URLs. Must exit immediately if the URL hash does not start with `#rc2jdt`.

Structure (IIFE, `var` declarations, `'use strict'`):

1. **Hash gate**: `if (!location.hash.startsWith('#rc2jdt')) return;` -- this is the very first line inside the IIFE. The script matches `*://*/*` but only activates on pages the service worker navigates to with `#rc2jdt` hash.

2. **DOM replacement** (defense-in-depth from MV2 rc2Contentscript.js pattern):
   - Strategy 1: Immediately call `document.open()` then use `document.close()`. After closing, build placeholder DOM using `createElement`/`appendChild` only (no inline scripts). Create a body with a "Loading CAPTCHA solver..." message, styled with `background: #3c686f; color: #fff; font-family: Arial, sans-serif; padding: 32px`.
   - Note: `document.open()`/`document.close()` is the proven MV2 pattern for complete DOM replacement. It removes the original page's CSP meta tags and all existing content. This is a deliberate architectural choice for security (replacing untrusted page content with our controlled DOM), not an XSS vector -- we control all content written to the document.
   - Strategy 2: On `readystatechange`, call a `clearDocument()` function that removes all child nodes from all `<html>` children (head, body)
   - Strategy 3: On `DOMContentLoaded`, remove any body elements that don't have `id="myjd-captcha-body"`

3. **Read job data**: `chrome.storage.session.get('myjd_captcha_job', function(result) { ... })` -- if no job found, show error message in body. If job found, call `renderCaptchaWidget(job)`.

4. **renderCaptchaWidget(job)** function:
   - Clear document head and body children using DOM manipulation (removeChild loops)
   - Set `document.title = 'CAPTCHA - ' + (job.hoster || 'JDownloader')`
   - Create body with `id="myjd-captcha-body"` and styling (centered, max-width 600px, padding 32px, background #f5f5f5)
   - Create header `<h2>` with "CAPTCHA for {hoster}" text
   - Create `<div id="captchaContainer">` for the widget
   - Determine CAPTCHA type:
     - If `job.captchaType` contains 'hcaptcha' or 'HCaptcha': create `<div class="h-captcha" data-sitekey="JOB_SITEKEY">`, load `https://hcaptcha.com/1/api.js` as external `<script>` element via `createElement('script')`
     - Else (reCAPTCHA): create `<div class="g-recaptcha" data-sitekey="JOB_SITEKEY">`, if `job.siteKeyType === 'INVISIBLE'` add `data-size="invisible"`, load `https://www.google.com/recaptcha/api.js` as external `<script>` element via `createElement('script')`
   - After widget div, inject skip buttons (reuse same UI pattern as localhost: 4 buttons, blue/white theme, event delegation, dataset.skipType). The hoster label comes from `job.hoster`.
   - After skip buttons, inject countdown timer (same 5-minute pattern as localhost)
   - Start token polling (same pattern: 500ms, check g-recaptcha-response and h-captcha-response textareas)
   - For invisible/v3 CAPTCHAs (`job.siteKeyType === 'INVISIBLE'`): after the external script loads (use `script.onload`), send a message to the service worker requesting MAIN world execution: `chrome.runtime.sendMessage({ action: 'myjd-captcha-execute', data: { siteKey: job.siteKey, v3action: job.v3action || '' } })`

5. **Token submission for MYJD flow**: When token detected by polling, send: `chrome.runtime.sendMessage({ action: 'captcha-solved', data: { token: token, callbackUrl: 'MYJD', captchaId: job.captchaId } })`
   - Note: The service worker routes this to my.jdownloader.org tabs via chrome.tabs.sendMessage, where webinterfaceEnhancer.js forwards it to the web interface's AngularJS context. The web interface then calls myjdDeviceClientFactory.sendRequest("/captcha/solve", ...) to submit the token to JDownloader via the MYJD cloud API. This routing IS the myjdDeviceClientFactory path per CONTEXT.md.

6. **Skip for MYJD flow**: Skip buttons send: `chrome.runtime.sendMessage({ action: 'captcha-skip', data: { callbackUrl: 'MYJD', captchaId: job.captchaId, skipType: skipType } })`

7. **Countdown timeout**: Same 5-minute timer, sends skip(single) on expiry with `callbackUrl: 'MYJD'` and `captchaId: job.captchaId`

8. **beforeunload cleanup**: Clear all intervals

- Body: styled message explaining user needs to log in via extension popup
