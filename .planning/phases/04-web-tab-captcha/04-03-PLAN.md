Modify `Rc2Service.js` to use the new MYJD CAPTCHA flow. The key change is in `onWebInterfaceCaptchaJobFound` -- instead of calling the old `onNewCaptchaAvailable` which tried to use rc2TabUpdateCallbacks, it now sends a message to the service worker to prepare the CAPTCHA tab.

1. **Update `onWebInterfaceCaptchaJobFound(tab, captchaData, captchaJob)`** (line 54):
   - Build the job details object (keep existing param extraction):
     ```javascript
     let jobDetails = {
         captchaId: captchaData.id,
         captchaType: captchaData.challengeType || captchaData.type,
         hoster: captchaData.hoster,
         siteKey: captchaJob.siteKey,
         siteKeyType: captchaJob.type,
         v3action: captchaJob.v3Action,
         targetUrl: captchaJob.siteUrl || captchaJob.contextUrl,
         callbackUrl: 'MYJD'
     };
     ```
   - Send to service worker: `chrome.runtime.sendMessage({ action: 'myjd-prepare-captcha-tab', data: { tabId: tab, jobDetails: jobDetails } });`
   - Remove the old `onNewCaptchaAvailable(tab, "MYJD", params)` call

2. **Simplify `onNewCaptchaAvailable(tabId, callbackUrl, params)`** (line 226):
   - This function is no longer called for MYJD flow (that goes through service worker now)
   - It may still be called from the `myjdrc2:captcha-new` ExtensionMessagingService listener (line 220)
   - For the `captcha-new` listener: if `callbackUrl === 'MYJD'`, the old code forwards to my.jdownloader.org tabs. This path is used when the MYJD web interface directly sends a captcha-new message (not through the `#rc2jdt` URL detection). Keep this path but update it to use the service worker message:
     ```javascript
     if (callbackUrl === 'MYJD') {
         // Route through service worker for MYJD flow
         chrome.runtime.sendMessage({
             action: 'myjd-prepare-captcha-tab',
             data: { tabId: tabId, jobDetails: { ...params, callbackUrl: 'MYJD' } }
         });
     }
     ```
   - For localhost callbackUrl: this path is not used (localhost CAPTCHA pages auto-activate via manifest content script). Can leave the function body mostly empty for localhost.

3. **Clean up unused code**:
   - The `rc2TabUpdateCallbacks` object is no longer used for MYJD CAPTCHA flow. It was previously used to store callback functions that ran when a tab finished loading. The new flow uses session storage instead. Keep the variable declaration (it may be referenced elsewhere) but it should no longer be populated for MYJD CAPTCHAs.
   - The `ExtensionMessagingService.addListener("myjdrc2", "captcha-get", ...)` handler (line 339) used the old `rc2TabUpdateCallbacks` pattern. This handler is for the old flow where the content script would ask for CAPTCHA params. The new flow uses session storage, so this handler can remain as-is (it will simply log "no captcha available" if called, which is fine).

4. **Ensure the `chrome.tabs.onUpdated` listener** (line 297) still works correctly:
   - It already detects `#rc2jdt&c=` hash and queries the API
   - It calls `onWebInterfaceCaptchaJobFound(tabId, captchaData.data, captchaJobResponse.data)` which now uses the new flow
   - It calls `onLoginNeeded(tabId)` if not connected -- this still works (redirects to loginNeeded.html)
   - No changes needed to this listener

5. **Keep existing protocol callback listeners** (canClose, loaded, mouse-move) -- these are for the OLD MV2 messaging flow through ExtensionMessagingService. They will remain for backward compatibility but the new localhost flow uses direct XHR from the content script (Plan 01). The MYJD flow does not use these callbacks.
